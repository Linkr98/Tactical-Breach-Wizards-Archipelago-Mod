This document is a **self-contained handoff** for building the Python `.apworld` (the
generation/server side) in a **separate repository**. The in-game client (a BepInEx/Harmony
C# mod) lives in another repo and is being built in parallel. The two halves communicate over
the standard Archipelago network protocol and **must agree on the frozen item/location id
contract described here**.

You do **not** need the game, the mod, or the original chat to build the apworld. Everything
you need is in this file plus the two data files you should copy alongside it.

---

## 0. Files to copy into the apworld repo

From the mod repo, copy these into your apworld project (e.g. a `data/` folder):

- **`ap_data.py`** — importable Python tables (`BASE`, `PLAYABLE_CHARACTERS`, `ITEMS`, `LOCATIONS`). **This is the canonical id source — build `item_name_to_id` / `location_name_to_id` directly from it.**
- **`ap_data.json`** — the same data plus **rich logic metadata** (per-mission acts, character usage, recruit info, goal, mission order). **Use this for region/rule logic.**

Both are generated by `tools/generate_ap_data.py` from an in-game data dump. **Never hand-edit
them and never renumber existing ids** — seeds depend on id stability. New content is appended
only.

## 1. Game identity & id space

- **Game name (exact string, must match the client):** `Tactical Breach Wizards`
- **Base id:** `271828000`. All ids are unique offsets from this. Item-id space and
  location-id space are independent (AP allows numeric overlap; they never actually collide here).
- The client checks locations **by id** and receives items **by id**. Your apworld's
  name→id maps must use the exact `(name, id)` pairs from `ap_data.py`. As long as you build
  the maps from that file, the two halves are guaranteed consistent.

## 3. Items (82 distinct codes)

Each entry in `ITEMS` is `(name, id, key, category, classification)`. The `key` is a stable
internal handle (e.g. `char:NavySeer`) — use it for cross-referencing logic; use `name` for the
AP item name.

| Category | Count | Classification | Meaning |
|---|---|---|---|
| `character` | 5 | progression | Unlocks a wizard. Place exactly 1 each. |
| `mission_access` | 29 | progression | Grants access to one campaign mission. Place exactly 1 each. |
| `ability` | 14 | progression / useful | 4 PerkUnlock "dream" specials (always progression — count toward perk gating) + 10 base-kit abilities (AP-unlocked; `useful` unless a requirement references them, then `progression`). SeerFinaleKnockBackDummy removed. Place exactly 1 each. |
| `perk_point` | 5 | progression | Perk-tree ability upgrades. **One code per character**, placed in copies equal to that wizard's real tree size: Zan 7, Jen 5, Banks 5, Dall 5, Rion 4 = **26 total** (see §6). Progression because they gate completion/goal checks (perk-power gating). |
| `outfit` | 23 | filler | Cosmetic outfit. Place exactly 1 each. |
| `confidence` | 5 | progression | Per-character "Confidence Boost". The mod blocks natural confidence earning, so these are the ONLY confidence source (spent at the outfit shop = the checks). Each worth `confidence_per_boost` confidence; place enough per wizard to afford all their outfits (see §6). |
| `filler` | 1 | filler | "Donut" generic junk (no in-game effect). Place as many copies as needed to fill the pool. |

Map classifications to `ItemClassification`: `progression` → `progression`, `useful` →
`useful`, `filler` → `filler`.

## 5. Logic / gating model (decisions are FIRM unless marked "recommended")

**Overall shape: loosely ordered.** The campaign is *not* a strict chain. The goal is that
late, hard missions aren't reachable before the player has accumulated power (characters,
abilities, perk points). Keep logic simple.

**Firm rules (from the project owner):**
- **Loose ordering** — do not force strict mission-by-mission sequence.
- **Random starting hero** — the player starts with **one random `character` item** plus **1 perk point on that wizard** (head-start; eases generation), via `start_inventory` / precollected.
- **Fixed scripted opening** — `Game_Prologue` ("The Asset") is ALWAYS the precollected start
  mission, and the `Complete: The Asset` location is **locked** to `Mission Access: Rushwater
  Reunion` (`Game_Witch_Intro`), so everyone plays the two scripted tutorials in order before
  the real AP. These two are scripted (they supply their own cast), so they require **no
  characters** in logic regardless of the random hero.
- **Recruitment missions must NOT require the character they recruit** — it's acceptable for a
  character to be unrecruited for the single mission where they'd be recruited. (Use each
  mission's `recruits` field to subtract that character from any computed requirement.)
- **Anxiety dreams require their wizard** — each `Game_Lucid_Dream_<Name>` mission requires the
  character it's about (Jen/Banks/Dall/Rion/Zan dream → that wizard unlocked).
- **Perk-power gating (best-wizard / per-slot)** — `recommendedPerks` per mission is the strongest
  single wizard expected there (NOT a team total), because a wizard can be fielded in multiple
  slots, so the binding constraint is your best wizard. A wizard's perk power = their perk-point
  items + 1 if their dream ability is owned (max: Zan 7 [no dream perk — SeerFinale removed],
  Jen/Banks/Dall 6, Rion 5). Gate = best UNLOCKED
  wizard's perk power ≥ ceil(percent × recommendedPerks). Confidence goals gate from act 1 (0-idx;
  "Act 2"+) via `goal_perk_percent`; completions ALSO from act 1 ("Act 2"+) via
  `completion_perk_percent`. Act 0 ("Act 1") ungated.
- **Team-wide ability gating (`requiredTotalAbilities` / `squadSize`)** — confidence goals
  that need many abilities at once (e.g. `AbilitiesInOneTurnGoal`, "use N abilities in one
  turn") carry a `requiredTotalAbilities` count on their location: auto-read from the level
  file's `AbilitiesInOneTurnGoalAbilitiesNeeded` (see `TOTAL_ABILITY_GOAL_PARAMS`), raisable
  per goal via `GOAL_REQUIREMENTS`' `"totalAbilities"`. Every confidence goal also carries
  `squadSize` — how many player wizards its room actually fields (distinct player-wizard
  prefabs in the .lvl; "Melee Wizard" is Jen; 0 = unparsed, apworld falls back to 4). The
  apworld requires the best possible squad — the room's `squadSize` best UNLOCKED wizards,
  each fielding 4 abilities minus any of their BASE-KIT ability items still missing (dream
  specials are upgrades to a base ability, not extra unique abilities, so they don't count)
  — to total ≥ the required count. Because the count can need ANY base-kit item, all
  base-kit abilities are classified progression while any such gate exists.
- **Victory** = check the location `mission:Game_Finale_Roof` ("Counterheist: The Roof").
  See `goal` in `ap_data.json`.

**Recommended logic (you may tune):**
- One **region per mission**, reachable when the player has that mission's `mission_access`
  item **and** satisfies the loose act-gate below.
- **Loose act-gate** (scales power before late missions): require a minimum number of
  collected `character` items (and/or completed earlier missions) to reach later acts
  (acts are 0-indexed) — e.g. act 1 → ≥2 characters, act 2 → ≥3, act 3 (finale) → all 5
  characters; act 0 has no gate. Expose the thresholds as options if you want them tunable.
- `confidence_goal` locations require their mission's region **+** the goal's `character`
  (a goal is earned by playing that specific wizard). Skip the requirement only if that
  character is the mission's own recruit (per the firm rule above).
- `outfit_purchase` locations require the outfit's `character` AND enough confidence, which
  now comes only from `confidence` items (the mod blocks natural earning). Use a per-character
  **staircase**: sort that wizard's outfits by `cost`; the i-th needs total confidence ≥ the
  cumulative cost, i.e. `≥ ceil(cumulative_cost / confidence_per_boost)` of that wizard's
  Confidence Boosts. Place enough boosts per wizard to cover their full wardrobe (see §6).
- `ability_unlock` locations: each is the reward for completing the anxiety dream in its
  `missionStageID` (e.g. `Unlock_ChainShockSuperchain` <- `Game_Lucid_Dream_Jen`), so place it
  in that dream's region — it inherits the dream's gate (dream access + that wizard + act).
- **Finale region** (act 3 / the three `Finale - *` missions, ending at `Game_Finale_Roof`):
  require all 5 characters + their `mission_access` items.

## 6. Item pool composition (must total exactly 215 = number of locations)

- Mandatory progression (place exactly one each): 5 `character` + 29 `mission_access` +
  5 `ability` = **39**.
- `outfit` filler: **23** (one each).
- `perk_point`: **exactly each wizard's real perk-tree size** — Zan 7, Jen 5, Banks 5,
  Dall 5, Rion 4 = **26 total**, always included as **progression** (they gate completion/goal
  checks). The game caps how many perk points a character can spend (natural leveling is
  suppressed by the mod), so placing more would be unspendable dead items.
- `confidence` (progression): per wizard, `ceil(total_outfit_cost / confidence_per_boost)`
  copies, so the whole wardrobe is affordable. Per-wizard outfit-cost sums: Zan 50, Jen 60,
  Banks 36, Dall 23, Rion 12. At the default `confidence_per_boost=5` that's 10/12/8/5/3 = **38**.
- `filler` ("Donut"): fill the **remainder** so the pool size equals 216.
  Default (perk on, `confidence_per_boost=5`): `216 − 37 (mandatory after 2 precollected) − 23
  − 26 − 38 = 92` donuts. `confidence_per_boost` must be ≥ 2 (at 1 the boosts can't fit).
- The random starting character is precollected, so remove **one** `character` item from the
  pool (place 4 of the 5 in the world) and give the 5th via `start_inventory`. Likewise the
  starting mission's `mission_access` is precollected/placed at start.

## 7. Options to expose (suggested yaml options)

- `confidence_per_boost` (int, default 5, min 2) — confidence granted per Confidence Boost
  item; sets the shop economy granularity and how many boosts get placed (see §6).
- `completion_perk_percent` (int 0-100, default 25) — best-wizard perk gate for completion (0-idx act 1 / "Act 2"+).
- `goal_perk_percent` (int 0-100, default 50) — best-wizard perk gate for confidence goals (0-idx act 1 / "Act 2"+).
- `death_link` (bool, default false) — optional; TBW is turn-based, so "death" semantics are
  TBD (e.g. a wizard downed, or a mission failed). Safe to ship off by default.
- (optional) act-gate thresholds if you want the loose ordering tunable.

## 8. Client ↔ apworld contract

- **slot_data** sent on connect:
  `{"death_link": <bool>, "goal": "finale_roof", "confidence_per_boost": <int>,
  "start_character": <internal>, "start_mission": <stageID>}`. `confidence_per_boost` is
  required so the client knows how much confidence to grant per Confidence Boost item. The
  client also learns precollected items through the normal **ReceivedItems** flow.
- The client reports completion by sending a **location check** for `mission:Game_Finale_Roof`;
  it also sets the AP **goal/Completed** status. You only need to mark that location as the
  victory condition in logic.
- Item delivery: the client applies items purely by **item id** (it suppresses the vanilla
  unlock and flips state when the matching id arrives). No special payload beyond standard AP
  item packets is required.

- Target a recent Archipelago API version; follow current apworld conventions.

## 10. Regenerating data (reference only — the mod repo owns this)

`ap_data.py` / `ap_data.json` are produced by `tools/generate_ap_data.py` from a runtime dump
of the game. If the contract ever needs to grow, it is regenerated there (append-only) and the
new files are copied over. The apworld should treat these files as read-only inputs.
